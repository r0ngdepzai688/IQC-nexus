# Task 002B — Proposed Synthetic Personnel Schema Decisions

## Status and scope

- **Readiness:** NOT READY.
- **Purpose:** decision proposal for Task 002B implementation; this document does not authorize implementation or release.
- **Data rule:** all examples and future fixture records must be invented, use `SYN-` identifiers and `example.invalid` addresses, and must not be derived from personnel artifacts.
- **Compatibility proposal:** if approved, adapters would map the proposed canonical fixture into existing frontend, seeder, entity and API contracts; Task 002B must not change those contracts.

## 1. Proposed canonical synthetic schema

If approved, the proposed canonical source would contain records with the following logical contract. Field names below are proposed fixture-level names, not an API or database change.

| Field | Type | Required | Rule |
|---|---|---:|---|
| `employeeId` | string | Yes | Unique, immutable, `^SYN-[A-Z0-9-]+$`; never copied from a real identifier. |
| `fullName` | string | Yes | Invented display name; Unicode preserved as UTF-8. |
| `preferredName` | string or null | No | Invented short display name; null when absent. |
| `department` | string | Yes | Synthetic organizational label. |
| `organization` | string | Yes | Synthetic organizational label. |
| `clName` | string | Yes | Synthetic classification/team label required by the current seed contract. |
| `knoxId` | string or null | No | Synthetic alias only; unique when present. |
| `email` | string or null | No | Lowercase and restricted to `example.invalid`. |
| `position` | string | Yes | Existing business-role/position vocabulary only. |
| `scope` | string | Yes | Existing authorization/display scope vocabulary only. |
| `systemRole` | `Administrator` or `User` | Yes | Values observed in `AuthContext.tsx`, `UserSeeder.cs` and user-management UI. Requiring an explicit value and defaulting to least privilege are proposals. |
| `dashboardProfile` | string | Yes | `Auto` is observed in `User.cs`, `UserSeeder.cs` and user-management UI. Using it as the proposed canonical default still requires approval. |
| `accountStatus` | `Active`, `Inactive`, `Pending`, or `Locked` | Yes | Values observed in `AuthContext.tsx` and user-management UI; backend `User.cs` directly defaults `AccountStatus` to `Active`. Requiring an explicit fixture value is a proposal. |
| `isActive` | boolean | Yes | Field observed in backend code. Deriving it as true only for `Active` is **Proposed; requires repository verification and approval before READY.** |
| `roleProfile` | string | Yes | `Standard` is **Proposed; requires repository verification and approval before READY.** No matching `RoleProfile` vocabulary/default was observed in repository code. |
| `part` | string or null | No | Synthetic part/scope label where needed. |
| `avatar` | string or null | No | Null unless a repository-local synthetic asset is approved. |
| `notes` | string or null | No | Synthetic test text only; must not contain secrets or personal data. |

If this proposal is approved, fixture metadata such as passwords, numeric database IDs and timestamps would not be treated as personnel identity fields. They would be generated by the backend adapter or seeder according to the approved defaults below.

## 4. Proposed defaults

| Area | Proposed default/policy |
|---|---|
| Authentication | If approved, generate a valid password hash during seeding and never store a plaintext password in the proposed canonical personnel fixture. |
| System role | `User` and `Administrator` are observed values. Defaulting proposed records to `User`, and requiring explicit minimal administrator records, are proposals requiring approval. |
| Business role | `Staff` is an observed value and existing API fallback. Using it as a proposed canonical default is **Proposed; requires repository verification and approval before READY.** Proposed canonical records would state `position` explicitly. |
| Role relationship | Resolve the existing `User` role by stable role name; fail validation if the required role is unavailable rather than silently assigning elevated access. |
| Dashboard profile | `Auto` is an observed value and existing backend default. Selecting it for the proposed canonical source requires approval. |
| Role profile | `Standard` is **Proposed; requires repository verification and approval before READY.** |
| Account status | All four named values are observed in frontend code and `Active` is the backend default. Using `Active` for proposed happy-path records and explicit non-active records for coverage requires approval. |
| Active flag | `accountStatus == "Active"` is **Proposed; requires repository verification and approval before READY.** Contradictory combinations would be invalid under the proposal. |
| Created timestamp | Set by the seeding operation in UTC; do not hard-code identifying or historical timestamps. |
| Updated/last-login timestamps | Null for newly seeded synthetic users unless a test explicitly requires synthetic lifecycle timestamps. |
| Created/updated by | Fixed non-personal service marker such as `synthetic-fixture`; never a real employee identifier. |
| Numeric ID | Database-generated; never canonical fixture identity. |
| Avatar and notes | Null/empty at persistence boundary according to the policy below; no external personal image or biographical text. |

## 5. Null and empty-string policy

- Proposed canonical fixture: if approved, required strings must be trimmed, non-empty values; optional values use JSON `null`, not `""`.
- Frontend adapter: preserve optional fields as `undefined` only where the current TypeScript contract requires it; do not convert missing required values into display fallbacks during validation.
- Backend adapter: convert optional nulls to empty strings only for existing non-nullable entity properties. This is a compatibility conversion, not canonical meaning.
- API projection: preserve current controller fallbacks and formatting; Task 002B must not change response semantics.
- Whitespace-only values are invalid for both required and optional fields.

## 6. Duplicate and invalid-record policy

- Reject the complete fixture before seeding on duplicate `employeeId`, duplicate non-null `knoxId`, malformed synthetic IDs, or non-`example.invalid` email domains.
- Compare identifiers using the same normalization and case-sensitivity chosen for the existing datastore; the implementation must document and test that choice before it is accepted.
- Do not silently keep the first duplicate, overwrite an existing user, or partially seed a rejected fixture.
- Reject missing required fields, unknown enum values, contradictory `accountStatus`/`isActive`, invalid email syntax, and strings containing control characters.
- Vietnamese and Korean Unicode must round-trip without mojibake. Unicode normalization behavior must be tested; names must not be used as unique identifiers.
- Duplicate and malformed examples belong in isolated negative tests, not the runtime happy-path fixture.
- Validation diagnostics must identify record position and field without logging credentials or copying sensitive source data.

## 7. Synthetic credential strategy

- If approved, the proposed canonical personnel fixture would contain no plaintext password and no reusable production credential.
- Development/test credentials must come from an approved local secret/configuration mechanism, outside Git, and be hashed by the existing backend password-hash implementation at seed time.
- Use one documented synthetic credential policy per environment; do not derive credentials from `employeeId`, name, email or any real-person attribute.
- CI should inject an ephemeral test value and must not print it. Production-like environments must disable or explicitly authorize synthetic credential seeding.
- Administrator fixture access must be opt-in and separately controlled. A committed static administrator password is prohibited.
