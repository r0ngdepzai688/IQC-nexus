# Master Plan adaptive import contract

## Supported input

Data Hub accepts non-empty `.xlsx`, `.xlsm`, and `.xls` workbooks up to 50 MB. It scans the first 50 rows and first 50 used columns, evaluates header depths one through three, propagates merged-cell parents, and detects the first data row independently. Only completely empty rows, repeated header rows, and explicitly recognized total/end-of-report footers are skipped. Every other partially populated business row is staged so missing required values become visible validation errors. Unknown columns are shown during inspection and otherwise ignored.

Canonical fields are `ProjectName`, `Basic`, `Area`, `Grade`, `Cat`, `QtyLprLqv`, `QtyLsr`, `PvrTarget`, `PraTarget`, `SraTarget`, `HwPic`, `MainLprLqvDate`, and `MainLsrDate`. `ProjectName`, `Basic`, `Grade`, and `Cat` are required. A missing or ambiguous required mapping produces `Failed - Missing Columns` with zero staging rows. `QtyLprLqv` is stored in the existing `QtyLpr` database column.

Header suggestions combine normalized aliases, approved learned mappings, fuzzy/token similarity, sampled data type, and weak depth/position tie-breaking. Inspection returns parent, child, effective path, sample values, detected type, confidence, and evidence. A mapping is learned only when the submitted mapping has explicit confirmation. Conflicting corrections increment rejection evidence. Failed imports do not persist learned mappings.

## Business identity and validation

The import identity is normalized `Basic + Cat`; SKU is not required, validated, or consulted by this workflow.

- Basic: Unicode NFKC, trim, collapse whitespace, uppercase for comparison.
- Cat: Unicode NFKC, trim, collapse whitespace, uppercase.
- Blank Basic or Cat never forms a valid key.
- `LCV` remains `LCV`; it is not converted to `LQV`.
- Duplicate normalized keys within one batch are blocking validation errors. The same Basic with different Cat values is allowed.

Core rows persist `BasicKey` and `CatKey`, with an unfiltered unique index on the pair. The legacy schema has no trustworthy Cat source, so the migration deliberately fails when any pre-existing MasterPlan row has the newly added empty Cat. It never infers Cat from SKU and never leaves legacy rows outside the identity constraint. Repository owners must first classify or remove legacy rows through a separately reviewed preparatory data migration, then rerun this migration. On the successful clean path, all persisted keys are generated by the shared C# Unicode NFKC normalizer; SQLite does not approximate that algorithm.

Hard row errors are blank ProjectName, Basic, Grade, or Cat and duplicate normalized keys. Invalid optional dates, chronology problems, unknown PIC values, and unresolved optional mappings require review. Excel date values may be native dates, OLE Automation serials, or parseable invariant text.

States are `ReadyToInsert`, `ReadyToUpdate`, `SkipNoChange`, `ValidationError`, `ReviewRequired`, `Blocked`, `Skipped`, and `Committed`.

## Reviewed upsert

- No core match: `ReadyToInsert`.
- Match with no mutable difference: `SkipNoChange`.
- Match with differences: `ReviewRequired`, conflict type `ExistingBusinessKey`, with old/new values for every changed field.

Mutable fields are ProjectName, Area, Grade, both quantities, PVR/PRA/SRA targets, HW PIC, and both main dates. Basic and Cat are immutable identity fields. Existing-key actions are `Update`, `Skip`, or `Cancel`, per row or for all pending existing keys. Update produces `ReadyToUpdate`; Skip preserves core; Cancel leaves the whole batch uncommitted. The legacy SKU-named resolution endpoint remains as a compatibility alias for Skip/Cancel, but new clients use `/api/DataHub/resolve-existing-business-key/{batchId}`.

Commit revalidates every normalized key inside one database transaction. It inserts `ReadyToInsert`, updates `ReadyToUpdate`, ignores `SkipNoChange`/`Skipped`, updates milestones, audit rows, and batch counters atomically, and rolls everything back on failure. Staging captures the target ID/version; a changed version causes a review-required concurrency failure. Unique-index races return an actionable Basic+Cat conflict rather than a generic server error.

Each updated field creates a `DataHubAuditLog` entry with batch, normalized identity, field, old/new values, actor, timestamp, and reason `Master Plan Import`. Counters distinguish inserted, updated, skipped, no-change, validation-error, and unresolved-review rows.

## Master Plan API and UI

The backend records query—not only React—returns rows where Grade contains `B` case-insensitively and Cat is `LPR` or `LQV` case-insensitively. It orders PVR target ascending with null last, then Basic and Cat deterministically. The UI displays Project Name, Basic, Area, Grade, Cat, LPR/LQV and LSR quantities, PVR/PRA/SRA targets, HW PIC, both main dates, status, and action.

## Legacy migration gate

Before deployment, run the migration against a copy of the target database. If `__MasterPlanLegacyClassificationCheck` fails, stop: the database contains MasterPlan rows for which this migration cannot assign Cat safely. Obtain an approved Basic-to-Cat classification and implement it as a separately reviewed preparatory migration, or explicitly retire those legacy rows under the repository's data-governance process. Do not infer Cat from SKU, delete records automatically, or weaken the unique constraint. Empty-database migration, the explicit legacy-row failure test, and the full test suite are the canonical pre-release checks.
